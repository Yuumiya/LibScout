Scraper — How It Works

  1. Search: Selenium drives a browser to GitHub search, extracts repo owner/name
  2. Download: download_repo() calls the GitHub API /repos/{owner}/{name}/tarball to fetch a tarball.
  3. Extract: Tarball is extracted to a temp directory; (RepoRef)
  4. Walk: RepoRef.iter_files() yields FileRef objects
  5. Read: FileRef.fetch_text()

GitHub Rate Limits

  • Crawling dozens of repos burns through the rate limit budget fast
  • The Search API cap is a huge bottleneck — paginating large result sets requires sleeps and retries
  • GitHub's secondary rate limits (429 responses) can trigger without warning during burst traffic, halting a crawl entirely.

Mitigations & Alternatives

What we do now

  • Token-based auth (5K/hr)
  • Rate-limit detection in the scraper — graceful abort on 429
  • Tarball download instead of cloning (one API call per repo)
  • Inter-page sleep to avoid burst triggers

Under consideration

  • Third-party providers — GH Archive, Libraries.io, or SourceGraph APIs that mirror GitHub data with more generous limits
  • Local git clone with shallow depth as a fallback when API limits are exhausted
GitHub's rate limits are very strict for a crawler at scale. For production use, relying solely on the GitHub search with selenium is not viable — we will likely integrate a third-party data provider to supplement or replace direct selenium calls.

Parser — Tree-sitter

Tree-sitter is an incremental parsing library that produces a Concrete Syntax Tree (CST) — a lossless, typed representation of source code.

Why not regex?

  • Regex can't handle nested structures
  • No semantic understanding of code
  • Breaks on edge cases and multi-line constructs

Why tree-sitter?

  • Supports 248 languages via tree-sitter-language-pack
  • Produces a full typed CST with node types

Parsing Flow


from libscout_parser import parse_file

result = parse_file("holonomy/__init__.py")

result.language      # → "python"
result.root_node     # → <Node type=module, children=6>
result.s_expression  # → "(module (import_statement …) (function_definition …) …)"

What the CST Enables

With the full Concrete Syntax Tree we can:

  • Extract function signatures — names, parameters, return types
  • Identify library call sites — trimesh.creation.icosphere(…) is parsed as a typed call_expression with known object + method
  • Build embeddings from structured code fragments
  • Query across languages — same tree-sitter query syntax works for Python, Rust, TypeScript, Go, etc
Next step: Extract meaningful code chunks from the CST, embed them with a code model, and store in ChromaDB for semantic retrieval
